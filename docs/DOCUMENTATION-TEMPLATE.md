# 文件撰寫模板與規範

## 📝 如何請求建立規格文件

當你需要為某個功能建立規格文件時，可以這樣說：

### 標準格式

```
「[功能名稱]功能，整理成一份文件，Spec-[功能名稱].md。目的是為了日後維護需要，簡潔扼要」
```

### 範例

1. **認證系統**（已完成）
   ```
   「登入註冊功能，整理成一份文件，Spec-Auth.md。目的是為了日後維護需要，簡潔扼要」
   ```

2. **查詢功能**
   ```
   「查詢功能，整理成一份文件，Spec-Query.md。目的是為了日後維護需要，簡潔扼要」
   ```

3. **知識庫管理**
   ```
   「知識庫管理功能，整理成一份文件，Spec-Knowledge.md。目的是為了日後維護需要，簡潔扼要」
   ```

## 📋 文件標準格式

### 基本結構

```markdown
# [功能名稱]規格文件

## 📋 概述
- 簡短說明功能目的
- 核心架構圖（如適用）

## 🔐 [主要功能流程]
- 流程說明
- 關鍵程式碼片段
- 檔案位置

## 📝 [次要功能]
- 狀態說明
- 建議實作（如適用）

## 🛡️ [保護/驗證機制]
- 說明
- 邏輯

## 🔌 API 端點
- 表格列出所有 API

## 📁 檔案結構
- 清晰的目錄樹

## ⚙️ 環境變數
- 配置說明

## 🔄 [狀態/流程管理]
- 說明

## 🗄️ 資料庫結構
- Schema 說明（如適用）

## 🔧 維護重點
- 關鍵邏輯說明

## 🐛 常見問題
- FAQ

## 📚 相關文件
- 連結到其他文件
```

## ✨ 文件撰寫原則

### 1. 簡潔扼要
- ✅ 重點說明，避免冗長描述
- ✅ 使用表格、列表整理資訊
- ✅ 關鍵程式碼片段即可

### 2. 結構清晰
- ✅ 使用 emoji 標題（📋 🔐 📝 🛡️ 🔌 📁 ⚙️ 🔄 🗄️ 🔧 🐛 📚）
- ✅ 章節分明，易於查找
- ✅ 使用表格整理 API、配置等

### 3. 實用導向
- ✅ 包含維護重點
- ✅ 列出常見問題
- ✅ 提供檔案位置

### 4. 完整性
- ✅ 涵蓋前端、後端、資料庫（如適用）
- ✅ 包含環境變數配置
- ✅ 說明關鍵流程

## 📖 文件命名規範

### 規格文件（Spec）
- `Spec-[功能名稱].md`
- 範例：`Spec-Auth.md`, `Spec-Query.md`, `Spec-Knowledge.md`

### 實作文件（Implementation）
- `[功能名稱]-IMPLEMENTATION.md`
- 範例：`QUERY-FEATURE-IMPLEMENTATION.md`

### 流程文件（Flow）
- `[功能名稱]-FLOW-COMPLETE.md`
- 範例：`LOGIN-FLOW-COMPLETE.md`
